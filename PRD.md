# PRD — Product Requirements Document

**Проект:** Pre-flight валидатор тендерной заявки
**Версия:** v1.0 MVP
**Дата:** 2025-02-04

---

## Executive Summary

**Продукт:** Pre-flight валидатор тендерной заявки — micro-SaaS для тендерщиков, который проверяет заявки (44-ФЗ/223-ФЗ) на ошибки и несоответствия ДО подачи на ЭТП.

**Проблема:** Отклонения заявок из-за формальностей/несоответствий → потерянная выручка, штрафы, потраченное время.

**Решение:** Pre-flight чекер, который ловит 90% причин отклонений ДО подачи.

**Ценностное предложение:** "Не теряйте контракты из-за формальностей. Проверьте заявку перед подачей — бесплатно."

---

## Product Vision

**Миссия:** Помочь тендерщикам подавать заявки без отклонений по формальным причинам.

**Видение:** Стать стандартом де-факто для pre-flight проверки тендерных заявок в России.

**Долгосрочная цель:** 10 000 пользователей через 12 месяцев.

---

## Target Audience (ICP)

### ICP-1: Тендерный отдел поставщика (20-500 сотрудников)
- **Команда:** 1-5 тендерщиков + юрист "по необходимости"
- **Зрелость:** Уже участвовали, уже получали отклонения, понимают цену ошибок
- **KPI:** win-rate, маржинальность контрактов, скорость подготовки заявки, % отклонений "по формалке"
- **Бюджет:** Уже платят за поиск тендеров/аналитику (Контур, Тендерплан и т.д.)
- **Оффер:** "Сократите % отклонений по формалке с 15% до 2%. Экономия времени — 3 часа на каждую закупку."

### ICP-2: Тендерное агентство
- **KPI:** количество "доведенных" заявок, отсутствие отклонений, SLA по срокам, прибыль/клиент
- **Главная боль:** процессность, контроль качества, дедлайны, шаблоны, масштабирование
- **Оффер:** "Автоматический pre-flight чек. 100% доведенных заявок. Возможность масштабирования без потери качества."

## Functional Requirements

### FR-1: Загрузка документов
**FR-1.1:** Пользователь может загружать файлы через drag & drop
**FR-1.2:** Пользователь может загружать папку с документами
**FR-1.3:** Система поддерживает форматы: PDF, DOC, DOCX, XLS, XLSX, ZIP, SIG, XML, HTML, JPG, PNG
**FR-1.4:** Система проверяет размер каждого файла (≤ 100 МБ по умолчанию)
**FR-1.5:** Система отображает список загруженных файлов с флагами ошибок

### FR-2: Проверка анонимности
**FR-2.1:** Система сканирует текст документа на наличие логотипов, названий компании, ИНН, телефонов
**FR-2.2:** Система проверяет метаданные файлов на наличие информации об авторе
**FR-2.3:** Система формирует отчет с подсветкой найденной информации
**FR-2.4:** Система отображает отчет по анонимности в боковой панели

### FR-3: Проверка подписей
**FR-3.1:** Система проверяет наличие КЭП на всех документах
**FR-3.2:** Система проверяет срок действия сертификата КЭП
**FR-3.3:** Система проверяет, что подписант имеет право подписывать (директор или по доверенности)
**FR-3.4:** Система проверяет наличие и валидность доверенности
**FR-3.5:** Система проверяет свежесть доверенности на момент подачи

### FR-4: Проверка обязательных документов
**FR-4.1:** Система проверяет, что все обязательные документы для типа закупки загружены
**FR-4.2:** Система предупреждает, если загружен документ, который не требуется по ТЗ
**FR-4.3:** Система проверяет, что выписка из ЕГРЮЛ не старше 6 месяцев
**FR-4.4:** Система проверяет, что НМЦК приемлема для поставщика

### FR-5: Проверка Формы 2
**FR-5.1:** Система проверяет, что все обязательные поля заполнены
**FR-5.2:** Система проверяет, что указана страна происхождения товара
**FR-5.3:** Система проверяет на расплывчатые формулировки ("от и до", "более и менее", "допускается", "должно быть")
**FR-5.4:** Система проверяет на противоречия между характеристиками

### FR-6: Дедлайн-менеджмент
**FR-6.1:** Система отображает время до конца подачи заявки
**FR-6.2:** Система предупреждает о техническом тайм-ауте (5-30 мин для обработки на ЭТП)
**FR-6.3:** Система отправляет напоминания за 24/6/1 час до дедлайна

### FR-7: Отчеты и чек-листы
**FR-7.1:** Система генерирует итоговую сводку (X критичных, Y предупреждений)
**FR-7.2:** Система генерирует детальный отчет по ошибкам (с описанием и исправлением)
**FR-7.3:** Система сортирует ошибки по критичности (BLOCK / WARN)
**FR-7.4:** Система отображает чек-лист "Перед загрузкой"
**FR-7.5:** Система отображает чек-лист "После Формы 2"
**FR-7.6:** Система отображает чек-лист "Перед подачей"

## Non-Functional Requirements

### NFR-1: Производительность
**NFR-1.1:** Время загрузки пакета документов (≤ 10 файлов) — ≤ 30 секунд
**NFR-1.2:** Время проверки одного документа — ≤ 5 секунд
**NFR-1.3:** Время генерации отчета — ≤ 10 секунд

### NFR-2: Usability
**NFR-2.1:** Пользователь может пройти весь процесс проверки за ≤ 5 минут
**NFR-2.2:** Интерфейс интуитивно понятен, без документации
**NFR-2.3:** Ошибки отображаются с понятным описанием и исправлением

### NFR-3: Security
**NFR-3.1:** Все данные шифруются в transit (HTTPS)
**NFR-3.2:** Пользовательские данные хранятся зашифрованно
**NFR-3.3:** Система не хранит загруженные документы после сессии (GDPR)

### NFR-4: Availability
**NFR-4.1:** Uptime ≥ 99.5%
**NFR-4.2:** Время отклика API — ≤ 200ms (p95)

### NFR-5: Compatibility
**NFR-5.1:** Система работает в Chrome, Firefox, Safari, Edge (последние 2 версии)
**NFR-5.2:** Система поддерживает iOS и Android (Mobile Safari, Chrome Mobile)

## Technical Architecture

### Technology Stack (предложение)

| Компонент | Технология | Почему |
|---|---|---|
| **Frontend** | React / Next.js | Популярно, есть специалисты, SSR |
| **Backend** | Node.js / Express | Совместимо с frontend, быстрая разработка |
| **Database** | PostgreSQL | Надежно, есть JSONB для rule-engine |
| **Rule-engine** | JSON schema + custom validation | Гибко, легко расширять |
| **Интеграции** | Axios / node-fetch | HTTP-запросы к API |
| **Хостинг** | Vercel / Railway | Быстро, бесплатно для start |
| **Мониторинг** | Sentry / LogRocket | Отслеживание ошибок |

### System Design

```
Frontend (Next.js)
├── Pages
│   ├── Landing page
│   ├── Загрузка документов
│   ├── Проверка (progress)
│   └── Отчет (results)
├── Components
│   ├── FileUploader (drag & drop)
│   ├── ErrorPanel (список ошибок)
│   └── Checklist (чек-листы)
└── API calls

Backend (Node.js/Express)
├── Routes
│   ├── /api/upload (загрузка файлов)
│   ├── /api/validate (проверка)
│   └── /api/report (отчет)
├── Services
│   ├── AnonymityCheckService
│   ├── SignatureCheckService
│   ├── RequiredDocsService
│   └── Form2ValidationService
├── Integrations
│   ├── CryptoProAPI
│   ├── EGRULAPI
│   └── ETPIAPI
└── RuleEngine
    ├── Rules (JSON schema)
    └── Validator

Database (PostgreSQL)
├── Tables
│   ├── users (пользователи)
│   ├── checks (проверки)
│   └── errors (ошибки)
└── Views (аналитика)
```

## User Stories

### US-1: Загрузка документов
**Как** тендерщик
**Я хочу** загрузить пакет документов для проверки
**Чтобы** убедиться, что все документы в порядке перед подачей

**Критерии успеха:**
- Могу загрузить файлы через drag & drop
- Вижу прогресс загрузки
- Могу загрузить папку с документами
- Система проверяет форматы и размеры

### US-2: Проверка анонимности
**Как** тендерщик
**Я хочу** проверить, что документы анонимны
**Чтобы** не отклонили заявку за наличие информации о компании

**Критерии успеха:**
- Система сканирует документы
- Вижу отчет с подсветкой найденной информации
- Вижу, что именно нужно исправить

### US-3: Проверка подписей
**Как** тендерщик
**Я хочу** проверить, что все документы подписаны и сертификат КЭП действителен
**Чтобы** не отклонили заявку за отсутствие подписи или просроченный сертификат

**Критерии успеха:**
- Система проверяет наличие КЭП на всех документах
- Система проверяет срок действия сертификата
- Система проверяет полномочия подписанта

### US-4: Проверка обязательных документов
**Как** тендерщик
**Я хочу** увидеть чек-лист обязательных документов
**Чтобы** не забыть загрузить важный документ

**Критерии успеха:**
- Система показывает чек-лист для типа закупки
- Система помечает, что загружено, а что нет
- Система предупреждает о лишних документах

### US-5: Проверка Формы 2
**Как** тендерщик
**Я хочу** проверить характеристики товара
**Чтобы** не допустить ошибки в Форме 2

**Критерии успеха:**
- Система проверяет обязательные поля
- Система проверяет страну происхождения
- Система ищет расплывчатые формулировки
- Система ищет противоречия

### US-6: Дедлайн-менеджмент
**Как** тендерщик
**Я хочу** видеть время до дедлайна
**Чтобы** не опоздать с подачей заявки

**Критерии успеха:**
- Вижу время до конца подачи
- Получаю напоминания за 24/6/1 час
- Вижу предупреждение о техническом тайм-ауте
